# ScopeServlet2 - 초심자부터 면접 대비까지 정리

## 1. 초심자용 실생활 비유
* **비유**: 바로 옆 창구로 안내해주는 **단순 안내원**
* **설명**: 
  - `ScopeServlet2`는 본인이 직접 서류를 검토하거나 정보를 추가하지 않고, 손님이 `/scope2` 창구로 찾아오면 즉시 해당 요청 서류를 그대로 들고 `scope.jsp` 담당자에게 안내(`forward`)해주는 단순한 중개인 역할을 수행합니다.
  - 이 안내원은 바인더에 데이터 포스트잇(`request.setAttribute`)을 붙이지 않고 넘겼기 때문에, JSP 담당자는 포스트잇이 없는 상태(Request attribute가 `null`)로 서류를 받게 됩니다.

---

## 2. 중급자용 실제 일어나는 일과 의존성, 문법 특성

### 단순 포워딩 컨트롤러(Forwarding Controller)의 특징
* 본 서블릿은 비즈니스 로직 처리나 데이터 바인딩(`request.setAttribute`) 과정 없이, 클라이언트의 GET 요청을 수신하는 즉시 `req.getRequestDispatcher("scope.jsp").forward(req, resp)`를 실행합니다.
* **Scope 데이터의 변화**:
  * 이 서블릿을 거쳐 `scope.jsp`가 렌더링될 때, `request` 영역에 새로운 속성이 설정되지 않았으므로 `${requestScope.action}` 이나 `<%= request.getAttribute("action") %>`은 `null`을 반환합니다.
  * 그러나 기존에 사용자가 브라우저를 통해 `ScopeServlet`을 거치면서 세션에 저장해둔 `"action"` 값(`session.getAttribute("action")`)이나 애플리케이션 영역에 저장된 공용 데이터는 세션과 어플리케이션이 활성화되어 있는 한 정상적으로 조회 및 출력이 가능합니다.

### MVC 패턴에서의 컨트롤러 역할
* 현업의 MVC 패턴 아키텍처에서는 보안상 브라우저가 JSP 파일의 실제 물리적 경로에 직접 접근하는 것을 막고, 반드시 컨트롤러(Servlet) 단을 거쳐 뷰(JSP) 파일이 렌더링되도록 통제하기 위해 이와 같은 포워딩 패턴을 적극 활용합니다.

## 3. 면접 준비를 위한 예상 질문

### Q1. `ScopeServlet2`를 거쳐 `scope.jsp`로 포워딩되었을 때, 기존에 `ScopeServlet`에서 세션에 저장한 `"action"` 값은 조회가 가능한가요? 그 이유는 무엇인가요?
* **A.** 네, 조회가 가능합니다. `ScopeServlet2`가 포워딩을 수행하더라도 브라우저(클라이언트)가 동일하다면 서블릿 컨테이너가 유지하는 `HttpSession` 객체는 파괴되지 않고 유지됩니다. 포워딩은 단지 서버 내부의 처리 흐름을 넘기는 것이며 세션 스코프의 데이터와 수명에는 영향을 미치지 않기 때문에 기존 세션 데이터를 정상적으로 조회할 수 있습니다.

### Q2. 로직이 존재하지 않고 단순히 JSP 페이지로 포워딩만 해주는 서블릿 클래스를 굳이 만들어 사용하는 이유는 무엇인가요?
* **A.** 두 가지 주요 이유가 있습니다.
  1. **보안성**: JSP 파일들을 외부에서 직접 URL로 접근하지 못하도록 `WEB-INF` 폴더 하위로 감추고, 반드시 서블릿 컨트롤러를 경유해서만 뷰에 접근할 수 있도록 URL 경로를 통제할 수 있습니다.
  2. **유지보수성 및 확장성**: 향후 페이지 접근 권한 체크(필터/인터셉터 등), 사전 데이터 로딩 등의 공통 전처리 로직이 필요해질 경우 JSP 코드를 수정할 필요 없이 해당 서블릿(컨트롤러) 코드나 설정만 수정하면 되므로 구조적 유연성을 확보할 수 있습니다.
