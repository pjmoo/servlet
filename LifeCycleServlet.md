# LifeCycleServlet - 초심자부터 면접 대비까지 정리

## 1. 초심자용 실생활 비유
* **비유**: 푸드트럭이나 팝업 스토어의 **단 한 명의 요리사**
* **설명**: 
  - **초기화 (`init()`)**: 이 요리사(`LifeCycleServlet` 객체)는 가게가 처음 오픈할 때 딱 한 번 채용되어 주방에 배치됩니다.
  - **요청 처리 (`service()`)**: 손님이 올 때마다 요리사가 새로 태어나는 것이 아니라, 이미 배치된 요리사 한 명이 계속해서 손님들의 주문을 처리합니다.
  - **동시성 이슈**: 요리사는 여러 명의 주문을 빠르게 처리하기 위해 **분신술(멀티스레드)**을 씁니다. 여러 손님이 한 번에 오면 분신들이 같은 주방 테이블을 공유하여 요리합니다.
    - 이때 주방 테이블 위에 공용 주문 횟수 계산기(`private int count`)가 있고, 각 분신들이 이 계산기를 조작한다면 서로 숫자를 덮어쓰고 혼동하는 대혼란(**Race Condition / 경쟁 상태**)이 발생합니다.
  - **소멸 (`destroy()`)**: 하루 영업을 완전히 마치고 가게를 닫을 때(서버 종료), 요리사는 주방을 정리하고 퇴근합니다.

## 2. 중급자용 실제 일어나는 일과 의존성, 문법 특성

### 서블릿의 생명주기 (Servlet Lifecycle)
1. **서블릿 인스턴스 생성 및 초기화 (`init()`)**:
   * 서블릿 컨테이너(WAS)는 서블릿 요청이 처음 들어오거나 설정에 의해 구동 시점에 서블릿 클래스를 로드하고 인스턴스를 생성(싱글톤)합니다.
   * 생성 직후 `init()` 메소드를 호출하여 데이터베이스 커넥션 풀 구축 등 무거운 리소스의 초기 설정을 수행합니다.
2. **요청 처리 (`service()` 및 HTTP 메소드 분기)**:
   * 클라이언트 요청 시마다 호출되며, 톰캣 내부 스레드 풀에서 할당된 개별 스레드 상에서 구동됩니다.
   * `HttpServlet.service()`가 내부적으로 HTTP 메소드 타입(GET, POST 등)을 식별하여 적절한 처리 메소드(`doGet()`, `doPost()` 등)로 분기시킵니다.
3. **서블릿 소멸 (`destroy()`)**:
   * 서블릿 컨테이너가 중지되거나 웹 애플리케이션이 언로드될 때 호출되어 열려 있는 파일, 자바 네트워크 커넥션 등을 안전하게 닫고 자원을 반납합니다.

### 싱글톤 구조와 스레드 안전성 (Thread-safety)
* 서블릿 컨테이너는 기본적으로 서블릿 클래스당 **단 하나의 인스턴스(싱글톤)**만 생성하여 공유합니다.
* 다수의 사용자가 동시에 동일한 서블릿으로 요청을 보내면, 컨테이너는 **멀티스레드(Multi-thread)** 방식으로 요청을 병렬 처리하며, 모든 스레드가 단일 서블릿 인스턴스를 공유합니다.
* 서블릿의 **멤버 변수(Field, Instance variable)**는 힙 영역에 존재하므로 모든 스레드가 동시에 접근해 변경할 수 있습니다. 
* 코드 내 `count++`와 같이 스레드 세이프하지 않은 연산이 포함되면 데이터가 오염되는 **경쟁 상태(Race Condition)**가 발생하므로, 실무에서는 서블릿 내부 멤버 변수를 사용하지 않는 **무상태(Stateless)** 설계가 강제됩니다.

## 3. 면접 준비를 위한 예상 질문

### Q1. 서블릿의 생명주기(Lifecycle) 단계와 각 메소드에 대해 설명해주세요.
* **A.** 서블릿은 `객체 생성 -> 초기화(init) -> 요청 처리(service -> doGet/doPost) -> 소멸(destroy)`의 단계를 거칩니다. `init()`은 최초 서블릿 로드 시 단 한 번 실행되며 리소스를 준비합니다. `service()`는 매 요청마다 스레드 풀에서 스레드를 할당받아 실행되며 HTTP 메소드별 분기를 수행합니다. `destroy()`는 컨테이너가 종료될 때 마지막 한 번 실행되며 자원을 해제합니다.

### Q2. 서블릿은 멀티스레드 환경에서 어떻게 작동하며, 멤버 변수 사용 시 어떤 문제가 발생할 수 있나요?
* **A.** 서블릿은 기본적으로 단 하나의 인스턴스만 생성되어 모든 요청 스레드가 공유하는 싱글톤 방식으로 동작합니다. 따라서 서블릿 내부에 공유 가능한 멤버 변수(상태 정보)를 두고 값을 변경할 경우, 여러 스레드가 동시에 접근하여 값을 오염시키는 동시성 이슈(Race Condition)가 발생할 수 있습니다. 이를 해결하기 위해 서블릿은 무상태(Stateless)로 설계하고 필요한 데이터는 지역 변수나 세션, 리퀘스트 객체 등을 활용해 관리해야 합니다.

### Q3. 하나의 서블릿에 여러 개의 URL 패턴을 매핑하고 싶을 때는 어떻게 해야 하나요?
* **A.** `@WebServlet` 어노테이션의 속성에 중괄호를 사용하여 String 배열 형태로 여러 URL 패턴을 지정할 수 있습니다. 예를 들어, `@WebServlet({"/lifecycle", "/lifecycle2"})`와 같이 설정하면 서로 다른 URL 주소 요청들을 단 하나의 서블릿 인스턴스에서 통합하여 처리할 수 있습니다.
